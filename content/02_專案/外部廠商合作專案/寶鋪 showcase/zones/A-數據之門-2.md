# A-數據之門-2 — 整合定稿（雙機全 TD）

> [!summary] 一句話結論
> **採 Plan C（雙機全 TD）為定案架構**：A 機（RTX 4080）扛三面牆 + 所有感測 + 導演時鐘；B 機（RTX 3070）扛地面 + 頂攝。兩機之間只傳「資料」不傳「畫面」。唯一前置關卡：D555 走乙太網路/DDS 進不了 TD，需用 Python→Spout 同機橋接（見 §14）。
>
> 本檔整合三份前置文件為單一權威來源：把 **Plan A 的實機勘查、可重用資產、外包決策、硬體待補規格**，與 **Plan B 的通信協議、warp 數學、延遲拆解**，全部收進 **Plan C 的雙機骨架**。

## 三案沿革（為何是 C）

| 案 | 來源檔 | 路線 | 動機 | 結局 |
|---|---|---|---|---|
| **A** | [[A-數據之門-TD搭建計畫]] | 純 TD、單機 | 留在 TD 生態、原生 mapping | 單機扛三牆+地面+重 shader 吃力 |
| **B** | [[A2-數據之門 plan b]] | Web-first、零 TD | TD 迭代慢；web 改存檔即見 | kiosk/維運隱性成本高、跨機同步靠自刻 |
| **C ✅** | （本檔） | 全 TD、**雙機分流** | 兩張卡兩台相機，拆開重渲染與輸出 | **定案**：留在 TD，但雙機求效能 |

> Plan B 的 web 路線本檔保留為「被否決的替代方案」與其技術洞見（warp/延遲），見 §11、§12 與附錄。

---

## 1. 設計原則

- 全程 TD，不引入 web。動機是**效能分流**：兩張卡、兩台相機，把重度渲染與輸出拆開。
- 不可分割單位：**面1/2/3 在「拉伸長螢幕」模式時是一張連續畫面，必須在同一台機器、同一 render context**，否則角落會出現接縫與不同步。→ 三面牆綁死在 A 機。
- 地面（面4）內容獨立、又吃不同座標系（頂攝），放 B 機剛好卸載。
- 機器之間只傳資料（OSC/CHOP），**絕不跨機串 TOP/NDI 影像**。
- 「會反覆改」的（七分鏡視覺、節奏、文案）放 TD network 迭代；「設一次」的（相機初始化、投影校正、kiosk/維運）不進迭代迴圈。

## 2. 雙機架構

```mermaid
graph LR
    D555[D555<br/>IR/depth + RGB] -->|Python+DDS→Spout 同機| A
    TOP[頂攝<br/>地面 XY] --> B
    ESP[ESP32 電容按鈕] -->|Serial / OSC| A
    MQTT[Welltek 12合一 MQTT] --> A

    subgraph A["A 機 · RTX 4080 · 牆面+感測+導演"]
        A[TD：拉伸長螢幕 render context<br/>狀態機/時鐘主控<br/>輪廓·數字·泡泡·集氣·爆白]
    end

    subgraph B["B 機 · RTX 3070 · 地面+頂攝"]
        B[TD：漣漪·掃描線·泡泡散落/收束<br/>頂攝 XY 追蹤]
    end

    A -->|OSC: scene/elapsed/trigger/sync| B
    B -->|OSC: floor XY| A

    A --> P1[面1 投影機]
    A --> P2[面2 投影機]
    A --> P3[面3 投影機]
    B --> P4[面4 地面投影機]
```

| | **A 機 — RTX 4080（強）** | **B 機 — RTX 3070（弱）** |
|---|---|---|
| 角色 | 牆面 + 感測 + 導演 | 地面 + 頂攝 |
| 相機 | **D555**（IR/depth 抽輪廓；微亮時用其 RGB 跑 InsightFace 抓年齡） | **頂攝**（地面 XY，blob/背景相減） |
| 投影輸出 | 面1 + 面2 + 面3（3 台，當一張寬畫布渲染再分區送出） | 面4 地面（1 台） |
| 渲染 | 牆面全部：粒子、門、波形、輪廓填數字、泡泡、溫度漸層、原子核集氣、爆白 | 地面：漣漪、掃描線、泡泡散落/收束 |
| 邏輯 | **狀態機 / 時鐘主控**、ESP32 觸碰、MQTT 環境數值 | 從 A 同步分鏡狀態 |

> 強卡吃最重的（三面牆 + 主角面所有重度 shader/particle + 一次性 InsightFace）；弱卡只扛一個地面輸出 + 很輕的頂攝追蹤，8GB VRAM 綽綽有餘。把牆全給 A 不只是負載，更是為了拉伸模式的連續性約束。

## 3. 投影面與相機系統

**投影面命名**：面1 左、面2 正前（中）、面3 右、面4 地。1+2+3 圍成 C 字形。
**牆面兩種模式**：A 拉伸長螢幕（三面當一張連續寬畫布）／ B 獨立（各牆各畫面）。

**兩台相機（D555 一機雙用）**

| 相機 | 機器 | 抓什麼 | 服務分鏡 |
|---|---|---|---|
| D555（depth/IR） | A | 人體正面輪廓 + 身高 | 3、4、5 |
| D555（RGB，同一台） | A | 微亮入場時抓年齡粗分桶／性別，存起來 | 供 4 暗場顯示 |
| 頂攝（一般相機） | B | 地面 XY 位置 | 1、2 |

**資料推算原則**
- 身高 → 用 D555 depth 幾何算（點雲垂直跨度 + 距離），暗場照樣準，不用 YOLO/pose。
- 年齡 → IR 暗場估不準；改在微亮入場用 D555 RGB 抓一次存起來，分鏡5 只需分小孩/大人/長者三桶。
- YOLO 在這版幾乎用不到（輪廓、身高被 depth 接走，年齡走入場 RGB），可整個拿掉省效能；除非要暗場多人偵測/persistent ID。

**8 項環境數值**（分鏡4/5）：CO2、HCHO、PM2.5、PM10、TVOC、光照、溫度、濕度 — Welltek 12 合一感測器（MQTT），進 A 機。

## 6. 狀態機（TD 實作骨架）

> 視角一/二是創意腳本；TD 內要落成離散狀態機 + 連續 phase。下表把 Plan A 的狀態機與本檔分鏡對位。

```
IDLE → AMBIENT → EXPLORE(phase 掃描→數據→個人化) → INTERACT(4.5→5→6) → FINALE(外包) → HANDOFF → IDLE
無人    進場/前言   分鏡1–4（含拉伸/獨立切換）        分鏡4.5–6 互動全壓面2   分鏡7         出口引導   回待機
```

| State | 對應分鏡 | 內容 | 來源 | 結束條件 |
|---|---|---|---|---|
| IDLE | — | 無人待機 | gen | 偵測到訪客（頂攝/輪廓面積） |
| AMBIENT | 1–2 | 進場粒子→門/前言/語音波形 | gen + 頂攝XY + audio | 平板 start / 停留 N 秒 |
| EXPLORE | 3–4 | 掃描人影 → 輪廓填數字 + 泡泡（8數據活）| gen + D555 + MQTT | phase 計時走完 |
| INTERACT | 4.5–6 | 按鈕淡入 → 觸碰溫度圖 → 集氣爆白 | gen + ESP32 + MQTT | 爆白動畫結束 |
| FINALE | 7 | 外包動畫（4面同步，TD cue 播放；地板我方補光）| video cue | 影片結束 |
| HANDOFF | — | 出口光門/箭頭引導 | gen | 訪客離場 → IDLE |

- 狀態機 = **A 機導演時鐘主控**，每幀廣播 `scene` 給 B；B 跟隨（見 §8）。
- EXPLORE/INTERACT 內用連續 phase 推進，不重建重物件。

## 8. 機器間通信（只傳資料，不傳畫面）

> [!warning] 絕不跨機串 TOP / NDI 影像
> 跨機傳影像要走 NDI，吃延遲與頻寬，是兩機 TD 卡頓的主因。所有需要的東西都用算好的數據在兩機各自重建。

- **A → B**：`/scene`（int）、`/scene_elapsed`（float）、`/trigger/*`、`/sync`（pulse）。OSC Out/In DAT，或 Touch Out/In CHOP。A 當時鐘主控、B 跟隨。
- **B → A**：`/floor/xy`（x, y）。分鏡1/2 牆面粒子要跟人在地面的位置。
- **地面泡泡同步**：分鏡5/6 地面與牆面泡泡要一致 → 不傳畫面，靠 A 廣播的 scene state + **兩機共用同一個 random seed**，各自算出相同分布。
- **同機內（A）**：D555 走 Spout、ESP32 走 Serial/OSC、MQTT 走 MQTT DAT。
- **既有接口**：實機 `oscin1` 已在 **port 7000** active（見 §13），OSC 命名應與 `visustwin.vision.detector` / `visustwin.mqtt.bridge` 對齊。

**資料 payload 規格**（沿用 Plan B 已定義的 schema，座標一律 normalized 0–1；輪廓送 polygon ~120–180 點，不送 pixel mask）：

```jsonc
// 相機來源（A 機 Python / B 機頂攝）→ 各自 TD，約 30fps
{
  "t": 1730000000.123,
  "presence": true,
  "zone": "in_position",                 // entering | in_position | at_wall | none
  "floor_xy": [0.48, 0.55],              // 頂攝（B）：地面 XY（分鏡1/2）
  "centroid": [0.51, 0.62],
  "bbox": [0.40, 0.20, 0.22, 0.70],
  "contour": [[0.41,0.22],[0.43,0.21]],  // D555（A）：分鏡3/4/5
  "height_cm": 168                       // depth 幾何算
}

// ESP32 → A
{ "id": "wall_btn_1", "state": "down", "t": 1730000000.4 }

// Welltek MQTT → A
{ "co2":612,"hcho":0.03,"pm25":14,"pm10":22,"tvoc":0.18,"lux":320,"temp":26.4,"humidity":58 }

// A 機導演 → B 機（每幀廣播 scene state，B 據此重建地面）
{
  "scene": 5, "scene_elapsed": 8.42,
```

## 9. 相機進 TD

- **D555 → A 機**：TD 的 RealSense 整合是 USB 取向，乙太網路/DDS 的 D555 大概率接不進去。務實解：A 機跑一支小 Python（pyrealsense2+DDS）抓 depth/IR/RGB，用 **Spout** 同機餵進 TD（同機 GPU 材質共享、零延遲，不是跨機，不違背全 TD 精神，只是 driver shim）。詳見 §14。
- **頂攝 → B 機**：一般相機，TD Video Device In + blob tracking/背景相減算 XY，OSC 給 A。

## 10. 跨機同步（閃白問題）

> [!important] 消費卡沒有硬體 genlock
> 真正的跨機 frame-lock 要 Quadro/RTX A 系 + Sync 卡，3070/4080 做不到。所以 A、B 之間的瞬切（分鏡6 爆白）理論上可能差一幀。問題本質是 **inter-display sync** 不是延遲，只有瞬間硬切才察覺得到。

- **牆與牆之間完美同步**：面1/2/3 同在 A 機、同 render context，無問題。
- **牆 vs 地（A vs B）的爆白**：做成 **3–4 幀（80–120ms）急速 ramp**，不要 1 幀硬切，1 幀差融進 ramp 看不出；且這是「牆 vs 地」垂直兩面、不是牆對牆連續，差一幀更難察覺。（元氣彈本就該有爆發曲線，零架構成本，九成機率問題消失。）
- **排程式觸發（備案）**：不廣播「現在翻白」，改廣播「在 timestamp T 翻白」（T 設未來 ~100ms），兩機等到各自時鐘的 T 才執行。對應 §8 的 `transition_at`。
- 分鏡2 衝向觀眾後高亮轉黑同理，用 ramp。

## 11. 投影 mapping

平面、一面一台、角落對接不重疊 → TD 原生 keystone/warp 直接做，這正是 TD 的強項（裝設期校一次、不進迭代迴圈）。

- A 機：三投影機當一張寬畫布渲染，再用 TD 分區 + 各自 keystone 送出，角落對接。
- B 機：地面單投影機 keystone。
- 不需要 edge blending（無重疊）。

> [!note] warp 的數學本質（Plan B 留下的洞見）
> 每面校正 = 一個 **homography**（四角透視映射），這就是 keystone 在做的事。一體拉伸模式 = 定義一張虛擬寬畫布（C 字攤平成一條），各區引用同一張內容空間 → 角落自然接上，粒子飄越面1→面2 是在同一空間連續移動。本案內容皆抽象流動（粒子/泡泡/漸層/波形），角落折痕看不出；只有「筆直線精準橫越 90° 角」才需真 3D projection mapping，本案無此需求。

## 12. 延遲

> [!important] depth 假設 30fps
> D555 走 Ethernet/DDS 時 depth 預設鎖 30Hz（issue #14284），60Hz 不一定拿得到（Viewer「解析度下拉鎖死」同一件事）。延遲預算用 30fps depth 估。

端到端 motion-to-photon 預算：

| 段落 | 量級 | 性質 |
|---|---|---|
| 相機曝光 + depth | ~33–66ms | 大頭 |
| D555 PoE/DDS 傳到 host | ~30–60ms（待實測） | 不確定 |
| 抽輪廓 + Spout | <5ms | 可忽略 |
| 機間 OSC（A↔B） | <10ms | 可忽略，非瓶頸 |
| render + vsync | ~16–33ms | 可優化 |
| 投影機 input lag | ~16–100ms+ | 隱藏大魔王，機型差異巨大 |

要點：
- 瓶頸在相機 fps、D555 PoE/DDS、投影機 input lag——**這些跟用不用 web 無關**，雙機 TD 一樣要吃。
- 投影機 input lag 是改善最大的單點：選低延遲機型 + 遊戲模式 + 關所有後處理（梯形/降噪/補幀都加延遲）。
- 輪廓對「站定後微動」容許度高；觸碰回饋走 ESP32 最快（無相機）。對輪廓做速度外推 + lerp 把延遲補回一部分。
- 抖動比絕對延遲更傷：鎖 fps、固定平滑窗口。

## 14. 相機源驗證：D555 over Ethernet → TD（最關鍵前置）

> [!danger] 唯一前置關卡
> D555 乙太網路/DDS 進 TD 不是現成路徑，要先打通 Python→Spout 橋。決定相機源是否成立。

- D555 走 Ethernet 官方支援（RealSense SDK 2.0 透過 DDS，原生對接 ROS 2）；現場已在 RealSense Viewer 跑起來，代表 Win11 上硬體 + DDS + 網路這條通。
- **風險在 Viewer → pyrealsense2 這步**：DDS 非預設開啟，wrapper 走 DDS 需 SDK 編譯加 `-DBUILD_WITH_DDS=ON`（一般 pip wheel 多半不含 DDS，裝了也抓不到乙太網路上的 D555）。可試 `pyrealsense2-beta`（Win wheel py3.10–3.13）。
- **已知未解 bug**（librealsense issue #14532，2025-12）：D555 只 USB 收得到、走 DDS 收不到，DDS wrapper 會「找不到任何 RealSense 裝置」，即使相機在網路上可見可達。
- **TD 當相機源的退路對乙太網路 D555 基本不成立**：TD RealSense 整合是 USB 取向，接不到 DDS 路徑。真正來源只有 pyrealsense2（開 DDS）或 ROS 2。
- 退路：撞上 DDS 找不到裝置 → 用 ROS 2 node 讀（D555 對 ROS 2 plug-and-play），再經本機橋 Spout 進 TD。
- 營運風險：RealSense 已從 Intel 切出由 RealSense AI 維護（repo `realsenseai/librealsense`），SDK 仍迭代、有未解 DDS bug。

**驗證路徑（建議入伍前或回來後第一件實測）**：
1. 從現場 Viewer 同一份 source 用 `-DBUILD_WITH_DDS=ON` + Python bindings 編出 pyrealsense2。
2. 跑最小 depth-grab 腳本確認拿得到幀 → Spout → TD 顯示確認。
3. 穩定 → 相機源定案。撞 bug → 退路 ROS 2 node。

**效能與穩定性要點**
- 雙機分流本身就是效能策略：A 扛重渲染+感測，B 扛地面+頂攝。
- TD Perform Mode 跑展演；關掉 Network editor 即時運算開銷。
- 兩機各自 watchdog 偵測心跳自動重啟；A 為時鐘主控，B 斷線重連後重新對齊 scene。
- 長跑注意 GPU memory，切分鏡不重建重物件；機間時鐘用 OSC sync pulse 對齊，避免 `scene_elapsed` 漂移。
