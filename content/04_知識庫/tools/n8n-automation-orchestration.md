# AI 自動化編排架構：n8n 確定性骨架 + 各單位訂閱 AI

> 本報告整理一次關於「如何讓 AI 穩定地做每日提醒與重複任務」的決策討論，
> 從個人需求一路推到中型組織的落地架構。結論可直接拿去規劃 VISION BASE Lab 的自動化基礎設施。

## 摘要（TL;DR）

最終定案一句話：

> **中央用 n8n 自架當「確定性骨架」（穩、社群大、好交接）+ 各單位自己的 AI 訂閱（成本可預測、不爆）+ 只有需要動腦時才呼叫 Claude；整套設定 + 流程定義用 GitHub 備份（憑證除外），可一鍵重建。**

這同時解掉所有痛點：

| 痛點 | 解法 |
|---|---|
| Claude 無法準時、主動推播提醒 | n8n 排程負責守時，Claude 只負責動腦 |
| 自主代理不穩 / 不用貴模型就變笨 | 模型當「乘客」只做窄任務，不當「司機」主導全程 |
| 規模化後 token 爆量 | 確定性骨架 + 訂閱制上限 + 快取 + 分層模型 |
| 機器要一直醒著 | 會議室 Mac mini，關電視照跑，一年電費 ≈ 一杯咖啡 |
| 重灌 / 交接就全沒了 | docker-compose + workflow JSON 進 GitHub，一鍵重建 |

## 1. 背景與問題

最初的問題是：**Claude 沒辦法很可靠地做「每日提醒」**。

原因不是 Claude 不夠聰明，而是它**缺一個「一直醒著、會主動推播」的身體**。Claude 能在排定時間「跑起來」，但它本身沒有穩定的常駐排程引擎，也不會主動把通知推到你眼前（除非你剛好在看那個介面）。

→ 結論：守時與推播這件事，要交給專門的工具，不該交給 LLM。

---

## 2. 核心觀念：自主代理 vs 工作流自動化

這是整份報告最關鍵的分水嶺。**差別在「誰在開車」。**

| | **自主代理**（Hermes / OpenClaw 類） | **工作流自動化**（n8n + Claude） |
|---|---|---|
| 誰決定流程 | **模型自己**邊想邊決定下一步 | **人**事先把流程畫死，固定 |
| 模型角色 | 司機（掌控全程） | 乘客（只在指定路段下車辦一件事） |
| 每次執行 | 可能都不一樣（非確定性） | 每次都一樣（確定性） |
| 穩定度 | 低（任一步想歪整條崩） | 高（骨架焊死，只有內容會變） |
| 模型需求 | 必須用貴模型，否則每個決策都選錯 | 便宜模型即可（任務窄、難度低） |

### 為什麼自主代理「不穩 + 燒貴模型」

- **不穩**：每一步都是模型臨場決定（要做什麼、呼叫哪個工具、要不要重試）。任一步幻覺、卡迴圈，整條就崩。執行一百次有一百種走法，無法預測。
- **燒貴模型**：模型要扛「規劃整個任務 + 做每個決策 + 自我糾錯」，需要極強推理力 → 笨模型在每個分叉都選錯 → 被迫一直用貴模型。

> 比喻：讓模型當司機，你得請最貴的司機還會出車禍；讓模型只當乘客辦一件小事，便宜的也辦得好。

### 但自主代理不是「比較差」

它們是為**開放式、你也說不清步驟**的任務設計的（「幫我研究這主題、自己想辦法搞定」）→ 靈活、有驚喜。
只是拿來做「每天準時、穩定觸發的提醒」=請一個天才即興演奏家當鬧鐘，他很厲害，但你永遠不知道明早他準不準時。

延伸閱讀：[[ai-frameworks/ai-frameworks|ai-frameworks]]、[[ai-frameworks/model-context-protocol|MCP]]

## 3. 正確的解：工作流自動化（n8n）+ 需要時才叫 Claude

```
n8n（穩定骨架：排程觸發 + 保證推播）
  └─ 需要動腦時 → 呼叫 Claude（或一個 agent）
       └─ 結果接回 n8n → 穩穩推給你（LINE / Email / Telegram）
```

各司其職：
- **守時** → n8n 排程引擎（它的本業，最成熟）
- **常駐** → 一台插電不睡的機器
- **推播** → LINE / Email / Telegram（服務本身極穩）
- **動腦** → Claude，只在那一格上場

可靠度評估（把失效情境都設好後 = production 等級）：

| 環節 | 穩定度 | 失效情境 & 解法 |
|---|---|---|
| 定時觸發 | ★★★★★ | n8n 本業，幾乎不錯 |
| 機器常駐 | ★★★★ | 停電/關機 → 設「復電自動開機 + 開機自啟」 |
| 推播送達 | ★★★★ | LINE/Email 本身極穩 |
| Claude 動腦 | ★★★★ | 偶發 API 逾時 → 流程設自動重試 |

加分：n8n 可設定「某次執行失敗時主動通知你」→ 連「靜默失敗」都能防。

## 4. 部署：中央大腦放哪

**結論：會議室 Mac mini 是最理想機型。**

### 關鍵觀念：螢幕關掉 ≠ 電腦睡眠

- 電視/螢幕關掉 → 只是 Mac mini 沒畫面輸出
- Mac mini 本身（桌機、插電）→ 照常運行，n8n 排程繼續跑

把「系統睡眠」設成永不即可：

```bash
sudo pmset -a sleep 0          # 系統永不睡眠（接電源時）
sudo pmset -a displaysleep 10  # 允許螢幕睡眠省電，但系統保持清醒
pmset -g                       # 確認設定
```

或 GUI：`系統設定 → 能源 →「螢幕關閉時避免自動進入睡眠」` + 「電力中斷後自動重新啟動」。

### 注意事項

- **不用佔整台**：n8n 跑在 Docker 背景，閒置 RAM 幾百 MB、CPU 趨近 0，平常照樣拿來做展示，互不干擾。
- **不一定要 24h 醒著**：Mac 可排程喚醒（`sudo pmset repeat wake MTWRFSU 08:55:00`），但因為 Mac mini 超省電（閒置 ~4–7W，一年電費約台幣 100–200 元），實務上直接讓它常醒最單純最穩。
- **HDMI 小坑**：少數電視關機後 Mac 會誤判沒接螢幕（通常無影響）；要百分百保險可插 HDMI dummy plug。
- **唯一前提**：那台機器要保持插電、別刻意關機（睡眠都設好不怕）。

### 不適合用筆電當中央大腦

筆電三個致命傷：①闔蓋就睡 ②會被帶走 ③綁個人。
→ 筆電適合「個人實驗 / 測試」；正式機構提醒用 Mac mini。兩邊同一套（Docker + n8n），流程匯出 JSON 就能搬。

## 5. 成本與 Token 經濟學

### 自架 vs 訂閱

| 方案 | 月費（約） | 性質 |
|---|---|---|
| **自架（自有機器）** | $0（+ 電費 ~$150/年） | 開源免費、功能全開；要自己準備常駐機器 |
| 自架（雲端 VPS） | US$5–10 | 機器在雲端永遠醒，跟學校電力/網路脫鉤 |
| n8n Cloud | US$20–24 起 | 免維護，有執行次數上限 |

VISION BASE 有現成 Mac mini → **自架最划算**，資料也留在機構內（研究機構的隱私加分）。

### 自主代理的成本（若採用）

- 純自主代理會瘋狂迴圈，**按 token 計費**帳單可怕又難預測。
- 若用 **Claude Code + Claude 訂閱方案（Max ~US$100–200/月）** → 吃到飽、成本固定可預測，對「會狂跑的代理」比 API 划算。
- 但對一人公司/小團隊，**混合架構（n8n 骨架 + 需要時叫模型）≈ US$30–80/月** 做同樣的事，更省更穩。

### Claude API 用量

每天一兩次小提醒 → 一個月幾十塊台幣，可忽略。

## 6. 規模化：中型組織如何不讓 token 爆量

**關鍵認知：token 爆不爆，取決於「架構」，不是「公司大小」。**

```
總 token ≈ 任務數 × 每次上下文大小 × 自主迴圈次數 × 使用人數
```

天真做法（給每人一個全自主代理）會在中型組織炸成天文數字。但同樣規模、同樣的事，靠架構可差 10–50 倍：

| 讓 token 爆炸 | 控制住 | 影響 |
|---|---|---|
| 自主代理全程自己推理、重試 | 確定性骨架編排，模型只做窄任務 | **最大** |
| 每次塞整份文件 | 只檢索需要的片段（好的 RAG） | 數倍 |
| 沒開快取 | **Prompt caching**（重複上下文只算一次） | 很大 |
| 什麼都用前沿貴模型 | 分層：簡單用便宜、難的才用貴 | 數倍 |
| 每人即時各跑各的 | 能批次的批次處理 | 一截 |

> 「公司大」只是放大量；「架構爛」才是把量再乘十倍的元兇。
> **正因為量大，中型組織更該用確定性架構，而非純自主代理。**

中型組織額外要管的三件事（一人公司不用煩）：
1. **成本治理（LLMOps / AI FinOps）** — 誰花多少、要不要設上限、可觀測性
2. **可靠性放大** — 出包影響人數變多，更需確定性骨架
3. **權限與資料邊界** — 誰的代理能讀哪些資料

## 7. 組織架構：中央鋪軌，各單位開車

完全分散（各單位自己從零搭）會養回筒倉問題（重複造輪子、無標準、綁個別單位）。
正解是**區分該集中 vs 該下放**：

| 層級 | 該集中（機構提供） | 該下放（各單位自管） |
|---|---|---|
| n8n 平台/骨架 | ✅ 共用基礎設施、流程範本庫、標準 | |
| AI 訂閱與額度 | | ✅ 各單位自己的訂閱、預算上限 |
| 憑證/資料權限 | ✅ 集中保管框架 | ✅ 各單位只碰自己的資料 |
| 成本可觀測性 | ✅ 機構總覽 | ✅ 單位自己也看得到 |
| 具體流程 | | ✅ 各單位建自己的自動化 |

```
        機構中央（平台層）
        ├─ 一套 n8n 基礎設施（共用、有人維護）
        ├─ 共用流程範本庫
        ├─ 統一成本/可靠性觀測
        └─ 標準與權限框架
                │ 提供「軌道」
        ┌───────┼───────┐
      單位A    單位B    單位C
   各自的AI訂閱 各自的預算 各自的流程
```

好處：訂閱獨立 → 成本歸屬乾淨、預算有上限、互不拖累；骨架與知識集中 → 不重複造輪子、有標準、人走流程還在。

> **誠實提醒：這套唯一的成本不是錢，是要有人當「平台維護者」**（顧中央 n8n、維護範本庫、盯成本）。中型組織約半個~一個人的職責。沒人扛，再好的架構半年後也會變沒人敢動的黑盒子。

## 8. 工具選型：n8n vs 替代品

### n8n 計費方式（重要）

n8n 是**按「工作流執行次數」計費，不是按節點或 token**。一次執行 = 整條流程跑一遍，不管 2 個還 50 個步驟都算 1 次。

| 方案 | 月費 | 含執行次數 | 超過怎麼辦 |
|---|---|---|---|
| Starter | ~$20 | 2,500/月 | **直接停**，等下月重置（無超額帳單） |
| Pro | ~$50 | 10,000/月 | 同上，直接停 |
| Business | ~$667 | 40,000/月 | 可超額，每 30 萬次約 €4,000 |
| Enterprise | 客製 | 無上限 | 不擔心超額 |

→ **一個每日提醒一個月才 ~30 次**。連 Starter（2,500 次）都等於可跑 ~80 條/天。小~中型組織根本碰不到上限，成本是零頭。Starter/Pro 超過就直接停、不爆帳單（壞處是真撞上限那天提醒會停 → 留餘裕或自架）。

### 替代品對照（依「AI 能搭 + 靈活 + 便宜 + 自架或訂閱皆可」篩）

| 工具 | 開源/自架 | AI 幫搭 | 便宜訂閱 | 特性 |
|---|---|---|---|---|
| **n8n** | ✅ 自架/雲端 | ✅ | ✅ | 最成熟、社群最大、平衡 |
| **Activepieces** | ✅ 開源自架/雲端 | ✅ | ✅ 更便宜 | 最像 n8n 的對手，成長最快，介面更乾淨 |
| Windmill | ✅ 開源自架 | 部分 | ✅ | 偏工程師（腳本+流程），效能快 |
| Make | ❌ 只有雲端 | ✅ | ✅ 有免費額度 | 視覺化最直覺，不能自架 |
| Pipedream | 偏雲端 | ✅ | ✅ 免費額度大 | 工程師友善，code + no-code |
| Gumloop | ❌ 雲端 | ✅ AI 原生 | ⚠️ 較貴 | 專為 LLM/代理工作流 |
| Zapier | ❌ 雲端 | ✅ | ❌ 貴 | 最易用、串接最多，但貴又不能自架 |

注意：AI 幫忙搭流程只能到 ~70%，剩下接線、憑證、邊界條件仍要人收尾。

### n8n vs Activepieces 社群規模（2026）

| | n8n | Activepieces |
|---|---|---|
| GitHub stars | ~186,000 | ~22,000 |
| 地位 | 霸主 | 成長最快的挑戰者 |
| 教學/範本/討論 | 海量 | 較少 |
| 找懂的人/外包 | 好找 | 難找 |

→ 差約 8 倍。對「中央骨架要長期穩定、好交接」的組織用途，**社群規模是被低估但關鍵的因素**（卡關有人救、人走好接手）。

### 選型結論

- **求穩、要好交接、海量資源** → **n8n**（VISION BASE 採用）
- Activepieces 留作「未來想省成本」的備案（概念相通，可搬）

## 9. 備份與災難復原（GitHub）

**鐵則：設定可以上 GitHub，憑證/金鑰絕對不行。**

| 東西 | 上 GitHub? | 做法 |
|---|---|---|
| `docker-compose.yml` | ✅ | 直接 commit |
| **workflow JSON（流程定義）** | ✅ | n8n 匯出 JSON → commit（重建保險） |
| 環境設定（非機密） | ✅ | commit |
| **API key / LINE token / 密碼** | ❌ 絕不 | 放 `.env` 加進 `.gitignore`，或用密碼管理器 |

兩種方式：
1. **手動匯出 → commit**（任何方案都能用，最簡單）
2. **n8n 內建 Git Source Control**（自動同步，但**自架版 / Enterprise 才有**）

與現況綜效（沿用 `visionbase-usc` org + Obsidian Git 同步習慣）：

```
visionbase-usc/
  ├─ vision_base_vault       (已有，知識/文件)
  └─ vb-automation (新開)     ← n8n 設定 + 流程備份
       ├─ docker-compose.yml
       ├─ workflows/*.json
       ├─ .env.example        (只列「需要哪些變數」，不放值)
       └─ .gitignore          (擋掉 .env)
```

→ Mac mini 就算被重灌：`git clone` + 放回 `.env` + `docker compose up`，幾分鐘完全復原，流程一條不漏。徹底解決重灌焦慮。

## 10. 結論與下一步

**定案架構：**

```
中央 Mac mini（會議室）
  ├─ Docker + n8n（確定性骨架：排程 + 編排 + 推播）
  ├─ 需要動腦 → 呼叫 Claude（窄任務、便宜模型優先）
  ├─ 推播 → LINE / Email / Telegram
  └─ 設定 + 流程 → GitHub 備份（憑證除外）

各單位
  └─ 自己的 AI 訂閱 + 預算上限 + 自己的流程（建在共用軌道上）
```

**落地路徑（從小到大）：**
1. 會議室 Mac mini 設好防睡眠 + 開機自啟
2. Docker 跑起 n8n，做出第一條每日提醒（推 LINE/Email）
3. 把 docker-compose + workflow JSON 進 `vb-automation` repo
4. 指定「平台維護者」角色
5. 逐步擴到各單位（共用範本庫 + 各自訂閱）

**待確認：**
- 第一個提醒推到哪（LINE / Telegram / Email）
- 第一個提醒的實際內容
- 誰擔任平台維護者
