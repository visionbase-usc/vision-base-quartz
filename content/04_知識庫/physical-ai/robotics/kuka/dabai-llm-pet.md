# physical-ai / robotics / kuka / 大白 LLM Pet

把一台 **KUKA KR 120 R2700** 變成有個性的「機械手臂寵物」——暱稱 **大白**。
打字跟它聊天,它用六軸做出**情緒/身體動作**回應;閒置時自己玩耍;動作由**本地 LLM 即時生成**,
甚至能**看貓的影片學動作**。建在實驗室既有的 `robot_web` 網頁 3D 介面之上。

- **GitHub**:[visionbase-usc/VB_kuka_KR120_r2700](https://github.com/visionbase-usc/VB_kuka_KR120_r2700)(目前在 `feat/pet-mode-llm` 分支)
- **本機**:`C:\Users\User\Documents\chrono\robot_web\`(前端 Three.js 免 npm + 後端 FastAPI)
- **執行環境**:conda `chrono` env + 本地 Ollama(見下)

> 定位:這是 [[kuka-loop-architecture-options|KUKA + AI loop]] 的一個**輕量、純前端視覺化**分支實驗——
> 不接真實控制器,而是探索「**LLM/VLM 如何生成機器人動作**」與「**如何讓工業臂看起來有生命**」。

---

## 1. 系統分層

| 層 | 角色 | 實作 |
|---|---|---|
| **大腦** | 意圖、情緒、個性、編動作 | 本地 LLM(Ollama `qwen2.5` 7B)|
| **眼睛** | 看影片→描述行為/節奏 | 本地 VLM(`qwen2.5vl:3b`)|
| **運動系統** | 把目標姿態變成流暢動作 | 前端「生命引擎」(臨界阻尼彈簧 + 呼吸層)|
| **身體** | 六軸 FK + 真實關節極限 | `robot_web` 既有 Three.js 模型 |

核心拆分原則:**「想做什麼」(LLM 擅長) 與「怎麼動得自然」(程序化引擎擅長) 分開**。

---

## 2. 動作生成管線

```
使用者訊息 ─▶ /api/chat ─▶ 本地 LLM ─▶ {reply, mood, motion}
                                          │
                          motion = 自編的關鍵影格 [[A1..A6 角度差°, 毫秒], ...]
                                          ▼
              後端 _sanitize_motion(夾限安全包絡 + 速度平滑 + 容錯)
                                          ▼
              前端生命引擎(彈簧追蹤 + Catmull-Rom 樣條 + 永遠呼吸)─▶ 六軸
```

- LLM 回 JSON `{reply, mood, motion}`;`motion` 是它**自己編的六軸關鍵影格**(無任何預設動作)。
- 後端 `_sanitize_motion`:夾限到溫和包絡 `[170,85,85,60,60,180]°`、**slew limiting**(拉長時間讓速度 ≤130°/s)、
  容錯(小模型常漏寫開頭的 A1=0 → **左補零**對齊)。
- 連不上 LLM → fallback 回離線關鍵字、**只回文字不動作**。

---

## 3. 生命引擎(讓工業臂「有生命」的關鍵)

逐格補間(tween 到關鍵影格、每格停死)本質上就是機械式。改用 **皮克斯 Luxo 跳跳燈** 的思路:

- **常駐迴圈 + 臨界阻尼彈簧**(ZETA≈0.62,輕微過衝)逼近目標 → 動量、緩入緩出、回彈**物理自然產生**,不會硬停。
- **always-on 生命層**:呼吸 + 重心微擺 + 頭部微動(不同頻率正弦疊加)→ **靜止時也在動**,不死板。
- **Catmull-Rom 樣條**串接關鍵影格 → 速度連續穿過,不停頓。
- **情緒→能量**:`mood` 推導 0~1 能量,調整生命層幅度(開心更彈、想睡更緩)。
- **不強制回原位**:依語意保持結束姿態(睡覺停低頭、好奇停歪頭);因每個影格目標都是「home+夾限差值」的**絕對姿態**,所以不會漂移。

## 4. 看影片學動作(VLM,可行性已驗證)

> 形態鴻溝:貓有四腳+柔軟脊椎+尾巴,大白是鎖在桌上的六軸臂 → **不可能複製骨架**。
> 能轉移的是**動作的氣質與節奏(timing)**,不是骨頭。

**兩段式管線**(各取所長,單一 VLM 直接吐數字會失敗):

1. **VLM 看影格 → 純文字描述**(`qwen2.5vl:3b`)。用自由散文,不要逼它填 JSON(小模型會 parrot 模板/吐亂碼數字)。中性 prompt 避免幻覺。
2. **文字 LLM(`qwen2.5` 7B)→ 把描述轉成六軸關鍵影格**。

實測一段橘貓理毛影片:VLM 準確描述「慵懶地舔尾巴、節奏緩慢」→ 7B 轉成慢動作 → 大白做出「慢轉身+偏頭理毛」。
再用 `cat_variants.py` 從同一描述生成多個變化 → `samples/cat_library.json` → 前端隨機循環播放。

## 5. 本地 LLM 環境(Ollama)

| 模型             | 用途            | 備註(RTX 3070 8GB)                     |
| -------------- | ------------- | ------------------------------------ |
| `qwen2.5`(7B)  | 對話 + 動作生成(預設) | 92% GPU/8% CPU 溢出,熱啟動 6–10s,理解身體指令較好 |
| `qwen2.5:3b`   | 較快替代          | 100% GPU,熱啟動 ~3.4s,但身體指令理解較弱         |
| `qwen2.5vl:3b` | 看影片描述         | 視覺;一次餵太多/太大張會慢,降到 4 張、384px          |

- 安裝:`winget install -e --id Ollama.Ollama`,服務開機自啟於 `127.0.0.1:11434`。
- 後端 `_call_local_llm`:`keep_alive=30m`(模型留記憶體)、`num_predict` 足夠容納影格。
- 抽影格:chrono env 裝 `imageio + imageio-ffmpeg`(自帶 ffmpeg,免裝系統 ffmpeg)。

## 6. 踩雷 / 經驗(可重用知識)

- **小模型做不了結構化數字**:VLM/LLM 對「受限數字陣列 + format:json」常退化(重複、亂填)。
  → 讓它**寫自然語言**,數字交給另一階段或程序化處理。
- **逐格 tween = 機械感**:要「有生命」靠的是**連續動量 + 永遠的微動(呼吸)**,不是更多關鍵影格。
- **大模型也會漏欄位**:7B 常漏寫開頭的 A1=0 → 解析要能**左補零/濾字串**容錯。
- **安全永遠在引擎層夾**:不論 LLM/VLM 產什麼,最終都夾限到真實關節極限 + 速度上限,亂數/幻覺都傷不了硬體(此處是純視覺化,但原則對真實控制更重要)。
- **conda 在 Windows**:`conda run` 遇非 ASCII stdout(cp950)會炸 → 設 `PYTHONUTF8=1`、輸出寫檔用 utf-8。

## 7. 現況與下一步

- **現況**:寵物模式可聊天、生成動作、閒置玩耍、聽懂「伸直/轉圈/鞠躬」等身體指令;VLM 看片學動作原型已通,有貓動作庫隨機循環(dev-hook 階段)。
- **下一步候選**:① 餵更多貓片擴充動作庫;② 做成正式「貓模式」UI 開關;③ 把學自影片的招式庫接回 LLM 聊天當「動作詞彙」由 LLM 即時組合。
